# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

#### Управление отоплением

- Создание новых датчиков температуры через REST API: `POST /api/v1/sensors`.
- Получение списка всех датчиков с актуальными показаниями температуры: `GET /api/v1/sensors`.
- Получение информации о конкретном датчике по ID: `GET /api/v1/sensors/:id`.
- Обновление параметров датчиков (имя, тип, расположение): `PUT /api/v1/sensors/:id`.
- Удаление ненужных датчиков: `DELETE /api/v1/sensors/:id`.
- Обновление текущих показаний датчика и его статуса: `PATCH /api/v1/sensors/:id/value`.
- Хранение конфигурации датчиков в PostgreSQL (имя, тип, расположение, единицы измерения).
- Автоматическое обновление показаний температурных датчиков из внешнего Temperature API при каждом запросе.
- Поддержка различных типов датчиков (в текущей реализации — temperature).
- Отслеживание статуса датчиков (активен/неактивен/ошибка).
- Фиксация временных меток последнего обновления показаний.

#### Мониторинг температуры

- Получение температуры для конкретного местоположения через внешний API: `GET /api/v1/sensors/temperature/:location`.
- Просмотр детальной информации о температуре: значение, единица измерения, статус, временная метка, описание.
- Получение актуальных данных о температуре в реальном времени из внешнего сервиса.
- Интеграция с внешним Temperature API для получения данных в реальном времени.
- Кэширование показаний в базе данных с возможностью ручного обновления.
- Обработка ошибок при недоступности внешнего API с логированием проблем.

#### Технические особенности

- Монолитное приложение на Go с использованием фреймворка Gin для REST API.
- PostgreSQL для хранения конфигурации датчиков и их истории.
- Асинхронное обновление температурных данных при запросе списка датчиков.
- Health check endpoint (`GET /health`) для мониторинга работоспособности.
- Конфигурация через переменные окружения (`DATABASE_URL`, `TEMPERATURE_API_URL`, `PORT`).
- Возврат расширенной информации о температуре (статус датчика, описание, временные метки).
- Таймаут запросов к внешнему API — 10 секунд.

### 2. Анализ архитектуры монолитного приложения

**Язык программирования:** Go с использованием фреймворка Gin для REST API.  
**База данных:** PostgreSQL для хранения конфигурации датчиков, их показаний и метаданных.

#### Архитектурные слои

- **Handlers** (`handlers/sensors.go`) — обработка HTTP‑запросов, валидация входных данных, формирование ответов.
- **Services** (`services/temperature_service.go`) — бизнес‑логика, интеграция с внешним Temperature API.
- **Models** (`models/sensor.go`) — структуры данных, DTO для создания и обновления датчиков.
- **Database** (`db/db.go`) — работа с PostgreSQL, CRUD‑операции.

#### Внешние зависимости

- **Temperature API** — внешний сервис для получения данных о температуре в реальном времени.
- **PostgreSQL** — база данных для хранения конфигурации.

#### Организация взаимодействия

- REST API с endpoints: `/api/v1/sensors`, `/api/v1/sensors/temperature/:location`.
- Синхронные HTTP‑запросы к внешнему API с таймаутом 10 секунд.
- Конфигурация через переменные окружения.

### 3. Определение доменов и границы контекстов

#### Выделенные домены

**Domain: Управление датчиками (Sensor Management)**

- **Сущности:** `Sensor`, `SensorType`, `SensorCreate`, `SensorUpdate`.
- **Ответственность:** CRUD‑операции с датчиками, хранение конфигурации.
- **Границы:** `handlers/sensors.go`, `models/sensor.go`, `db/db.go`.

**Domain: Мониторинг температуры (Temperature Monitoring)**

- **Сущности:** `TemperatureResponse`, `TemperatureService`.
- **Ответственность:** получение актуальных данных о температуре из внешнего API.
- **Границы:** `services/temperature_service.go`, `handlers.GetTemperatureByLocation`.

**Domain: External API Integration**

- **Сущности:** `TemperatureService`, HTTP‑клиент.
- **Ответственность:** взаимодействие с Temperature API, обработка ошибок, таймауты.
- **Границы:** `services/temperature_service.go`.

#### Bounded Context

- **Smart Home System** — единый контекст, объединяющий все домены в рамках монолита.

### 4. Проблемы монолитного решения

- **Единая точка отказа** — при падении монолита недоступны все функции: и управление датчиками, и мониторинг температуры.
- **Жёсткая связанность** — handlers напрямую зависят от `temperature_service` и `db`, что усложняет независимое масштабирование компонентов.
- **Отсутствие изоляции доменов** — все домены развёртываются вместе, невозможно обновлять или масштабировать их независимо.
- **Сложность масштабирования** — при росте нагрузки на мониторинг температуры приходится масштабировать весь монолит, а не только соответствующий сервис.
- **Ограниченная отказоустойчивость** — проблемы с внешним Temperature API влияют на всю систему (при запросе списка датчиков происходит синхронный вызов API).
- **Сложность технологического развития** — все компоненты привязаны к одному стеку (Go + Gin), невозможно использовать специализированные технологии для разных доменов.
- **Монолитная база данных** — PostgreSQL используется для всех данных, что создаёт риски блокировок и сложностей при миграции схемы.
- **Отсутствие явных границ контекстов** — в коде нет чёткого разделения между доменами, что затрудняет понимание и поддержку.

# Задание 4. Создание и документирование API

### 1. Тип API

Для взаимодействия микросервисов будет использоваться комбинация синхронных и асинхронных API:

**Синхронное взаимодействие:**

- **REST API (HTTP/HTTPS)** — для CRUD-операций и взаимодействия SPA с сервисами. REST выбран за простоту, универсальность и широкую поддержку во всех фреймворках (Gin, FastAPI, Express, Spring Boot). HTTP-запросы легко тестировать, отлаживать и кэшировать.
- **gRPC** — для высокопроизводительного сервис-сервисного взаимодействия (Device Management ↔ Telemetry, Telemetry ↔ Temperature Monitoring). Бинарный протокол Protocol Buffers эффективнее JSON для частого обмена данными, обеспечивает строгую типизацию контрактов и поддержку streaming.

**Асинхронное взаимодействие:**

- **HTTP webhooks** — для асинхронных уведомлений между сервисами. Webhooks проще в реализации и развёртывании по сравнению с брокерами сообщений (RabbitMQ), не требуют дополнительной инфраструктуры и легко поддерживаются всеми используемыми фреймворками.

**Обоснование выбора:**

- REST API обеспечивает простоту интеграции и отладки для внешних клиентов (SPA)
- gRPC оптимален для интенсивного внутреннего взаимодействия между сервисами
- HTTP webhooks позволяют реализовать событийную модель без операционных overhead брокеров сообщений
- Такой подход балансирует между простотой реализации и производительностью системы

### 2. Документация API

Документация API для каждого микросервиса представлена в формате OpenAPI 3.0 (Swagger):

#### Device Management Service (Go + Gin)

- **Спецификация**: [Device Management API - OpenAPI](docs/to-be/api/device-management-api.yaml)
- **Endpoints**:
  - `GET /api/v1/devices` — получение списка всех устройств
  - `GET /api/v1/devices/{id}` — получение устройства по ID
  - `POST /api/v1/devices` — создание нового устройства
  - `PUT /api/v1/devices/{id}` — обновление устройства
  - `DELETE /api/v1/devices/{id}` — удаление устройства
  - `GET /api/v1/devices/{id}/status` — получение статуса устройства

#### Telemetry Service (Python + FastAPI)

- **Спецификация**: [Telemetry API - OpenAPI](docs/to-be/api/telemetry-api.yaml)
- **Endpoints**:
  - `GET /api/v1/telemetry/{deviceId}` — получение телеметрии устройства
  - `GET /api/v1/telemetry/{deviceId}/history` — история показаний
  - `POST /api/v1/telemetry` — отправка показаний (от устройств)
  - `GET /api/v1/telemetry/stats` — статистика по устройствам
  - `WebSocket /ws/telemetry` — realtime обновления телеметрии

#### Temperature Monitoring Service (Node.js + Express)

- **Спецификация**: [Temperature Monitoring API - OpenAPI](docs/to-be/api/temperature-monitoring-api.yaml)
- **Endpoints**:
  - `GET /api/v1/temperature/{location}` — получение температуры по локации
  - `GET /api/v1/temperature/sensor/{sensorId}` — получение температуры по ID сенсора
  - `GET /api/v1/temperature/history/{location}` — история температур
  - `POST /api/v1/temperature/cache/refresh` — принудительное обновление кэша
  - `GET /api/v1/temperature/alerts` — получение температурных алертов

#### Notification Service (Java + Spring Boot)

- **Спецификация**: [Notification API - OpenAPI](docs/to-be/api/notification-api.yaml)
- **Endpoints**:
  - `GET /api/v1/notifications` — получение списка уведомлений
  - `POST /api/v1/notifications/templates` — создание шаблона уведомления
  - `PUT /api/v1/notifications/{id}/read` — отметка уведомления как прочитанного
  - `GET /api/v1/notifications/unread-count` — количество непрочитанных уведомлений

#### Асинхронные события (AsyncAPI)

- **Спецификация**: [Smart Home Events - AsyncAPI](docs/to-be/api/events-asyncapi.yaml)
- **События**:
  - `device.created` — создание нового устройства
  - `device.updated` — обновление устройства
  - `device.deleted` — удаление устройства
  - `telemetry.received` — получение телеметрии
  - `threshold.exceeded` — превышение порога температуры
  - `temperature.updated` — обновление данных о температуре
  - `alert.triggered` — срабатывание алерта
  - `alert.resolved` — разрешение алерта
  - `notification.send` — запрос на отправку уведомления
